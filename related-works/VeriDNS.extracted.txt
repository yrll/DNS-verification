VeriDNS: Incremental Distributed Verification of DNS
Configurations
anonymous author
Abstract
The Domain Name System (DNS) is a critical Internet component, yet it
remains vulnerable to misconfigurations that can cause widespread outages.
Existing centralised validation tools suffer from performance bottlenecks. In
contrast, distributed validation tools cannot support incremental validation,
making it struggle to handle both the scale and the high frequency of config-
uration updates common in modern infrastructures. In this work, we present
VeriDNS, a distributed framework designed to address these twin challenges.
VeriDNS models DNS resolution as a formalized resolution state graph, an
abstraction that is inherently suitable for parallel, distributed validation.
Crucially, this model enables highly efficient incremental verification: con-
figuration changes are translated into localized graph updates, eliminating
the need for costly full re-computations. By embedding formal semantics
into the validation process, VeriDNS ensures both syntactic and semantic
correctness. Our experimental results demonstrate that VeriDNS not only
detects a broader class of misconfigurations than state-of-the-art tools but
also achieves significant performance gains, showcasing the power of its dis-
tributed and incremental architecture.
1. Introduction
The Domain Name System (DNS), formalized in RFCs 1034 [21] and
1035 [22], serves as a cornerstone of the Internet’s infrastructure. Its function
as a scalable, hierarchical naming system that translates human-readable do-
main names into machine-routable IP addresses underpins virtually all global
digital services. However, the distributed architecture that grants DNS its re-
silience also renders it inherently susceptible to misconfigurations [23, 31, 34].
These errors, ranging from missing records to subtle aliasing and delegation

The persistence of these critical issues highlights a long-standing chal-

on utilities like “dig” [4] and “nslookup” [19] for ad-hoc analysis, while static

useful for catching formatting errors, these methods were fundamentally in-
capable of reasoning about the semantic behavior of the resolution process
across multiple, interacting zones. Their inability to detect logical errors left
a critical gap as the Internet’s complexity began to grow.
The manifest inadequacy of these syntax-focused tools necessitated a
paradigm shift towards more principled and exhaustive analysis. This led
to the development of formal verification methods, a significant leap from
passive detection to proactive, model-based correctness proofs. Tools such
as GR oot [14] centralized collection of all configuration files, spearheaded
this movement by systematically modeling the DNS resolution process. By
simulating all potential query paths, GRoot could proactively identify deep,
logical errors like inconsistent delegations and resolution loops before deploy-
ment, enabling a far more rigorous validation.
Yet, as the Internet’s scale and dynamism continued to expand, the very
exhaustiveness that made these formal methods powerful also revealed their
critical limitations. Built on a centralized data collection, tools like GR oot
encountered significant computational and scalability bottlenecks when an-
alyzing the vast, interdependent DNS zones. While subsequent research ex-
plored alternative directions—such as Maude, which focused on DDoS re-
silience [16], or Octopus [27], which proposed a distributed architecture—a
key challenge remained unaddressed. Octopus, despite tackling distributed
verification, did not offer an efficient solution for incremental verification. In
modern, dynamic environments where configurations are updated frequently,
the necessity of re-verifying the entire state from scratch for each minor
change renders such systems impractical.

This evolutionary path has brought the field to a critical juncture, leaving
fundamental challenges unsolved: scalability in distributed environments and
efficiency for handling frequent, incremental updates. The central question
thus becomes: How can we verify DNS configurations in a manner that is
both massively scalable and inherently incremental?
In this paper, we propose VeriDNS, a distributed verification framework
designed to answer this question. VeriDNS introduces a novel graph-based
abstraction of DNS resolution behavior, which formally represents the reso-
lution process as a resolution state graph. This model is not only inherently
suited for distributed computation but also naturally accommodates incre-
mental validation. By doing so, VeriDNS enables scalable, high-performance
verification.
Key Advancements Our approach introduces several key improvements
over the state of the art:
• Scalability: The VeriDNS model is optimized for distributed envi-
ronments, ensuring that large-scale DNS configurations can be verified
without the performance bottlenecks of centralized systems.
• Incremental Verification: Our graph-based framework naturally
supports incremental validation. Configuration changes translate to
localized updates on the graph, enabling re-verification at a microsec-
ond level.
• Results: Our experimental evaluation demonstrates that VeriDNS
finds a broader range of distributed errors than the centralized GRoot
and achieves superior performance than the distributed Octopus, par-
ticularly showcasing its powerful incremental verification capabilities.
Through this work, we provide a powerful and practical tool for ensuring the
correctness of DNS configurations in today’s complex and dynamic Internet
infrastructure.

Despite its seeming simplicity, the DNS resolution process is fundamen-

Misconfigurations are alarmingly frequent and can arise from manual er-
rors, outdated migration scripts, uncoordinated updates across authorita-
tive servers, and record misalignments in delegations. Such issues can re-
sult in name resolution inconsistencies, partial reachability, or complete ser-
vice outages. Notable outages at major providers like GitHub, Spotify, or
Cloudflare have been attributed to subtle but critical DNS misconfigura-
tions [3, 29, 26, 9, 17, 7].
Existing Tools. Monitoring-based tools often fail to uncover such bugs due
to the distributed and non-deterministic nature of resolution paths [32, 6, 25].
These tools, such as “dnschecker” [5] and “whatsmydns” [28], are designed to
check the global propagation of DNS records and monitor their availability
and performance from various locations. While valuable for observing the live
state of the DNS infrastructure, their reliance on periodic checks of specific
record types (e.g., A, MX) means they are reactive. They can identify when a
service is unreachable, but cannot proactively detect underlying logical flaws
in zone file configurations that might only manifest under specific, and often
rare, query patterns. Their core limitation is that they do not model the DNS

Static and formal analysis tools offer a more proactive approach by di-

environments where configurations are updated frequently, the necessity of
re-verifying the entire state from scratch for each minor change renders such
systems impractical and underscores the need for a new approach.
Formalizing Resolution as Graph. The core limitation of existing tools,
as discussed, is their failure to model the actual resolution process—a dy-
namic sequence of state transitions across distributed zones. To overcome
this, we argue for a fundamental shift in perspective: instead of viewing
zone files as static tables, we must model the query resolution path itself
as a traversal through a state-transition graph. This abstraction allows us
to capture the operational semantics of DNS behavior, which is essential for
uncovering complex, interactive bugs.
We formalize this concept as a resolution graph. In this graph, each node
represents a logical state in the resolution of a query ⟨d, t⟩, d is the query
name, t is the query type, and each directed edge represents a state transition
driven by a specific RR. The states can be categorized as follows:
• Initial state ( si): Where a resolver begins processing a query for a
given zone.

• Intermediate state ( sm): Represents a non-terminal step where the
query is rewritten or delegated via records like CNAME or NS.
• Final state ( sf): Where the resolution terminates, yielding a defini-
tive answer such as an IP address, or an authoritative NOERROR (with
no data) or NXDOMAIN response.
This graph-based formalization provides a powerful analytical framework.
It enables us to rigorously reason about critical properties of DNS configu-
rations that are otherwise elusive to monitoring or purely static approaches:
• Determinism: Do all valid resolution paths for the same query converge
to the same final state? Divergence can indicate serious misconfigura-
tions.
• Termination: Can any query enter a loop, thus never reaching a fi-
nal state? This identifies bugs like cyclic dependencies in CNAME or
DNAME records.
• Reachability: Are there valid records in a zone file that can never be
reached by any legitimate query, indicating dead or orphaned configu-
rations?
Analyzing these properties is critical for ensuring the reliability and security
of DNS, especially in distributed environments where zone files are managed
independently.
Crucially, this method is inherently suited to address the key challenges
of distribution and incremental updates that hobbled previous systems. By
decomposing the global resolution process into a collection of interconnected
per-zone subgraphs, it provides a natural foundation for distributed verifi-
cation. Furthermore, a change to a single RR corresponds to a localized
modification of the graph, enabling efficient incremental analysis without
re-verifying the entire state from scratch. This abstraction forms the cor-
nerstone of our proposed framework, VeriDNS, which we detail in the next
section.

3.1. Resolution State Graph
Zone files, being central to DNS configurations, are often represented in a
semi-structured format designed primarily for performance, but not for for-
mal reasoning or analysis. To bridge this gap and enable rigorous analysis,
we aim to model each RR in a zone file as a state transition in a formal state
dependency graph. This approach facilitates the use of graph-theoretic and
logical operations to analyze the interactions between the records. Inspired
by previous work [30, 33, 14], we define the RSG as a directed multi-graph,
where each edge represents a resolution or transition from one state to an-
other, modeling DNS resolution as a sequence of state transitions. This rep-
resentation allows us to reason about semantic properties, coordinate cross-
zone analysis, and efficiently analyze resolution behavior.
Formal Definition. We model a DNS zone as a set of RRs,Z = {R1, R2, . . . ,
Rn}. The resolution behavior within this zone is represented as a Resolution
State Graph (RSG), defined as a labeled, directed multi-graphG = (V, T, E),
where:
• V : A set of vertices (or states), where each vertex v ∈ V represents a
unique DNS entity, such as a domain name or a terminal value (e.g.,

• T : A set of transition types, corresponding to the different types of
DNS records (e.g., A, NS, CNAME).
• E: A set of directed edges. Each edge e = ( si, t, sj) ∈ E represents
a state transition, signifying that the entity si ∈ V can resolve to the
entity sj ∈ V via a RR of type t ∈ T .
Each RR in the zone is parsed into one or more edges in the RSG. For
example, an A record www.school.com. A 192.0.2.1 corresponds to an
edge ( www.school.com, A, 192.0.2.1). The resolution process is formal-
ized by a transition function δ : V × T → V , where δ(si, t) = sj indi-
cates that entity si resolves to entity sj via record type t. For instance,
δ(www.school.com, A) = 192.0.2.1. This model provides a rigorous math-
ematical foundation for analyzing DNS configurations while remaining intu-
itive through its graph-based structure.
Construction Process. Given a zone file Z, the corresponding RSG is
built through a structured process that distinguishes between internal and
external dependencies.
Intra-Zone Graph Expansion. The process starts by building the graph
from all RRs defined within the zone file Z. We initialize an empty graph
G = ( V, T, E) and iterate through each RR. For each record, parsed as a
triple (si, t, sj) representing (owner name, type, rdata): The source entity si
and target entity sj are added as vertices to V if they are not already present.
A directed edge ( si, t, sj) is added to E. This initial loop constructs a graph
that captures all explicit, locally-defined resolution paths within the zone.
Identifying External Dependencies. RRs such as NS records (delega-
tions) or CNAMEs pointing to external domains inherently create depen-
dencies on other zones. Our construction process identifies these but does
not attempt to resolve them at this stage. When a target entity sj (e.g.,
ns1.another-provider.com) is not defined within the current zone Z, the
vertex sj is marked as an external stub node. These stubs are critical connec-
tion points that serve as the interface for the cross-checking phase described
later. This clear separation prevents the construction process from overstep-
ping its bounds into inter-zone verification.
Graph Convergence The RSG is constructed iteratively by processing the
RRs in a DNS zone Z = {R1, R2, . . . , Rn}. A graph expansion operator F
adds vertices and edges to the RSG G = (V, T, E) based on the RRs. The

3.2. Property Verification
Once the RSG is constructed, we perform property verification to de-
tect semantic and structural issues in DNS configurations. This verification
process is divided into two complementary modes: self-checking and cross-
checking. Each mode is underpinned by rigorous formal semantics, which
ensure that the checks are both precise and automatable.
Self-checking. The self-checking phase is designed to validate the internal
correctness of a single DNS zone by performing local, semantics-aware anal-
ysis over its RSG. Given an RSG G = (V, T, E) constructed as described in
§3.1, we traverse this graph and use a suite of logical predicates to identify se-
mantic violations. These predicates are defined based on the graph structure
and capture common classes of DNS misconfigurations, such as CNAME
loops, missing glue records, and improper delegations. The following key
properties are checked via graph-based predicates:
• Missing Glue Record: For each NS delegation to a nameserver, if the
target nameserver’s domain name lies under the current origin and does
not have an accompanying A or AAAA record, a glue record is missing:
(si, NS, sj) ∈ E ∧ sj ∈ origin(z) ∧ ∄(sj, t, sk), t ∈ {A, AAAA}.
• Lame Delegation: If an NS record points to a nameserver that does
not respond authoritatively for the delegated zone, the delegation is
considered lame: ( si, NS, sj) ∈ E ∧ query(sj) = REFUSED.
• Rewrite Loop: A CNAME chain must eventually terminate in a non-
CNAME record. Let π = {(s1, CNAME, s2), . . . ,(sk,

Cross-checking. While self-checking ensures the internal consistency of a
single DNS zone, DNS is inherently distributed, and correctness depends not
only on individual zone files but also on the consistency of delegation chains
and glue records across zones. Many misconfigurations arise from interac-
tions across multiple zones maintained by different administrative entities.
The cross-checking phase targets such inter-zone issues by validating dele-
gation relationships and shared dependencies between zones. This process
leverages the previously constructed RSGs of multiple zones, applying inter-
zone predicates to identify delegation inconsistency, cyclic zone dependency,
rewrite loops, overflow, and rewrite blackholings.

Shared Path Reuse. To improve efficiency, our cross-checking process
reuses resolution paths computed during self-checking. We model the DNS
resolution process as a functional mapping, where multiple domain queries
may resolve to a single canonical endpoint. Suppose two queries, q1 (from
zone Z1) and q2 (from zone Z2), both resolve to the same canonical name, c,
for instance, through a chain of CNAME records. This relationship can be
expressed as: resolve(q1) = c and resolve(q2) = c. Given this many-
to-one mapping, the validation of both q1 and q2 becomes dependent on the
validation of the common target, c. Instead of independently performing
the full resolution and validation for each query, we introduce an optimiza-
tion. The resolution path and validation status of the canonical target c
are computed once. Consequently, any query that resolves to c can reuse
this cached validation result. This logic can be formally stated as follows:
If resolve(q) = c, then valid(c) = ⇒ valid(q). This optimization signifi-
cantly reduces query overhead and computational redundancy in large-scale
deployments, particularly where many domains redirect to a common canon-
ical target.
3.3. Incremental Verification
In modern operating environments, DNS configurations are not static but
are frequently updated to add services, change IP addresses, or modify dele-
gation policies. After each minor change, revalidating the entire configuration
from scratch is not only computationally expensive but also fails to provide
the timely feedback required. Therefore, an efficient incremental validation
mechanism is not only a performance optimization but also a critical feature
indispensable in actual deployments. The key insight behind our incremental
approach is that most configuration changes have a localized impact. The
explicit dependency structure of the RSG allows us to precisely identify and
isolate this impact. Instead of treating the zone file as an opaque block of
text, we can map a change to a specific set of vertices and edges in the graph
and then analyze how this local change propagates through the system.
Our incremental verification process for DNS zone modifications unfolds
in three distinct stages. First, any change to a zone file—whether an inser-
tion, deletion, or update of an RR—is translated into a set of elementary
graph operations, forming a delta set ∆ G. This set encapsulates the direct
modifications to the RSG. Second, to capture the full impact of these changes,
including cascading effects, the algorithm performs a comprehensive impact
analysis. It conducts a two-way graph traversal starting from the nodes in

targeting only the affected portions of the graph, this approach efficiently
verifies both direct and indirect consequences of the update with minimal
computational overhead, addressing the challenge of verifying dynamic DNS

Building the RSG. By reading the zone file line by line and converting
each line into a triple ( si, t, sj), we can construct the RSG. For example,
the record a○ is converted to ( campus.com, N S, ns1.campus.com). If the
state node is not present in the RSG, it is added to the graph. Then, an
edge labeled N S is added between campus.com and ns1.campus.com. Sim-
ilarly, for the record b○ converted to ( campus.com, N S, ns2.campus.com),
ns2.campus.com is added to the RSG and an N S edge is added between
campus.com and ns2.campus.com. By repeating these steps, a complete
RSG can be constructed.
Property verification. In the self-checking phase, the verifier initiates
a depth-first traversal from nodes with zero in-degree in the RSG. During
traversal, the verifier examines each path, along with its edge types and state
transitions, to check property-specific conditions. For example, in Figure 2,
we obtain the path ( campus.com, N S, ns2.campus.com), which satisfies the
condition glue(si, sm). If the next node of ns2.campus.com is empty, without
an A record or AAAA record, it indicates a missing glue record.
The cross-checking involves transforming the specific graph data accord-
ing to the property that needs to be checked. The necessary data is then
sent to the appropriate addresses for verification. For example, to check
for delegation inconsistency, glue-related data G1

extracted and sent to IP address “17.17.17.31”. The address “17.17.17.31”
receives the data and the property to be checked, and then performs property
verification using the local RSG data G2

Incremental Verification. Now, assume an administrator modifies record
d○ , changing the IP address for “ns2.campus.com” from “17.17.17.32” to
“17.17.17.33”. Instead of re-validating everything, VeriDNS’s incremental
engine takes over. The change is mapped to the deletion of the old A record
edge and the addition of a new one. From this focal point, the impact
analysis begins. A backward traversal from the “ns2.campus.com” vertex
immediately identifies the incoming NS record b○ from “campus.com” as an
affected predecessor path. A forward traversal from the new IP node finds
no further outgoing dependencies. The resulting affected subgraph is thus
precisely identified. The verifier then re-runs its checks only on this small
subgraph: the self-check confirms the glue record is still valid with the new
IP, and the cross-check is re-triggered, sending the updated delegation proof
to the relevant nameservers to ensure global consistency is maintained.
5. Evaluation
We evaluate the effectiveness and efficiency of VeriDNS through five key
questions: Q1. Can VeriDNS detect more DNS misconfigurations than ex-
isting methods? Q2. Is VeriDNS more efficient than centralized tools such
as GR oot in verifying configuration errors? Q3. Does the distributed
and reusable nature of VeriDNS significantly reduce verification overhead
at scale? Q4. How effective is incremental verification of the VeriDNS? To
answer these questions, we conduct four sets of experiments: single-device ex-
periments (§5.1) to evaluate the bug detection ability of self-checking, multi-
device experiments (§5.2) to measure the time efficiency of distributed veri-
fication, compare experiments (§5.3) to measure the verification reusability,
and incremental experiments (§5.4) to measure the efficiency of incremental
verification.

5.4. Incremental Experiments
Dataset. We use the same dataset as in §5.2. From this dataset, we select a
representative zone file containing over 10,000 RRs. To evaluate incremental
verification, we simulate updates by injecting 10,000 new RRs under three
scenarios: (1) A records, (2) CNAME records, and (3) randomly selected record
types. The objective is to assess whether VeriDNS can efficiently validate
changes without re-verifying the entire zone and to check whether errors
such as blackholing are introduced.
Results. Figure 8 summarizes the verification latency under the three
update scenarios. VeriDNS achieves microsecond-level performance across
all cases: 99% the verification time is ≤ 2.39 × 104 ns for A RR and ≤
3.16 × 104 ns for CNAME RR insertions, and ≤ 3.34 × 104 ns for random
updates. These results demonstrate that incremental validation remains
lightweight even under large-scale changes, confirming the efficiency of our
change-propagation mechanism.
Answer to Q4. Incremental verification in VeriDNS is highly effective.
By localizing validation to only the affected subgraph, the framework avoids
redundant recomputation and maintains near-constant latency regardless of
update type. This validates the practicality of VeriDNS for real-world DNS
environments where frequent updates are common.

6. Discussion
Scalability and Deployment. VeriDNS adopts a decentralized architec-
ture and leverages state graph abstraction to support scalable DNS configu-
ration validation. Our results ( §5.2) demonstrate that verification tasks can
be distributed across multiple devices without requiring centralized access
to all zone files. This allows VeriDNS to scale gracefully with large DNS
deployments, especially in multi-administrator environments such as CDNs,
academic networks, or enterprise IT infrastructures.
Formal Semantics Enable Precision. Unlike purely heuristic-based ap-
proaches, VeriDNS defines misconfiguration properties through formal se-
mantics. This rigor allows us to verify DNS behaviors with well-defined
correctness criteria, avoiding ambiguity or false positives. Furthermore, this
formalism enables future extension to support custom policy checks or new
record types (e.g., SVCB/HTTPS records).
Limitations and Future Work. While VeriDNS shows strong results,
there remain challenges. First, our current model assumes consistency in
DNS responses during verification time, which may not hold under rapidly
changing environments (e.g., short TTLs or load-balanced DNS). Second, al-
though our framework is compatible with most RFC-compliant zone files, cer-
tain edge cases such as DNSSEC validation, DoT [10], DoH [8], and DoQ [11],
interactions are currently out of scope.

In future work, we plan to explore: (1) deeper integration with DNSSEC

7. Conclusion
We present VeriDNS, a distributed framework that tackles the key chal-
lenges of scalability and incremental efficiency in large-scale DNS verifica-
tion. At its core, VeriDNS introduces a novel graph-based resolution state
abstraction that naturally decomposes the verification task for distributed
computation. This model maps configuration changes to localized graph up-
dates, enabling incremental re-verification at microsecond-level latency. Our
evaluation demonstrates that VeriDNS not only outperforms the centralized
GRoot in finding errors but also its verification performance significantly
surpasses existing distributed approaches like Octopus. Ultimately, VeriDNS
provides a powerful and practical solution for ensuring DNS correctness and
reliability in today’s large-scale, highly dynamic Internet environments.
